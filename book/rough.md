# 粗糙

现在的 MVP 比起五年十年前更难一些，必须没有短板，更加没有一招鲜的空间了

## 开始可以简单粗糙一点

读《小而美：持续盈利的经营法则》时，看到一段：

>整个软件是一份包含了 2700 行大部分是复制粘贴的代码的 Python 文件，存储在 Google 云端。（之后我已经将代码开源，链接见本章结尾。）但是它成功了！它解决了问题，所以我就推出了。当然，它还没有为大众“做好准备”。但是 10 年后，Gumroad 仍然感觉没有准备好，我认为它今后也不会。

>等等，没有付款功能？是的！我收集了每个人的 PayPal 信息。在每个月的月底，我会制作一份列出每个人邮件地址和他们账户余额的表单，然后一个一个地给他们付款。最后，我开始将流程一点一点地自动化。我写了一些代码来下载列表，而没有从数据库一行一行地复制粘贴。后来，我编写了一份指令，利用 PayPal 的 API（应用程序接口）发送付款。

知识星球早期倒没那么粗糙，但也有不少类似的例子。比如：

在推出收费能力的时候，我们完全没做甚至也没考虑应该怎么“续费”。当时的想法是：反正收费功能是按年——而且绝不能做短期的，这样至少能再熬九个月才开始琢磨续费这事儿。

再比如：

其实刚开始做的时候，完全没考虑并发（其实也不是没考虑，是就不太懂），于是当时 Fenng 发一个朋友圈/公众号/微博，召唤大家加他的圈子，我们就会当机一次。印象中，三次之后，慢慢地能力提升，再后来的大事故，就是生财有术的 418 了。

糙有糙的好处——不要过早优化，毕竟可能超过 99% 的产品，都没能熬到有 100 个并发连接？

## 粗糙

这几天跟同事聊天，继续提到了粗糙，我隐约觉得，我们可能有 1/4 到 1/3 的时间和精力，消耗在了过早地“进入细节”，甚至实现了细节。

之所以再提这事，是前段时间跟同事聊起知识星球的网页版改进，其中有个需求，是我提出的。我的需求点是：

我在网页版写内容，写完后，以往在其他网站的习惯，往往是按一个 tab 或在 command + Enter，就能发表，但在知识星球却做不到，希望能支持。

产品同事在看到这个需求后，又结合了其他人提过的需求，以及在网上看到的一些产品的快捷键支持情况，将需求整理为：

- n 打开发表框
- Ctrl + Enter 或 Command + Enter 发表
- Space 向下滚动一屏
- 上下键滚动
- l 点赞
- c 评论
- j 下一条主题
- k 上一条主题
- o 打开主题里的图片、文件、视频
- / 聚焦到搜索框

倒不是说这些需求不对，只是，这么一来，一件比较轻的事情，从设计到实现，都被放大了，时间也会消耗更多。

这就不得不提到 37signals 的 Shape Up 了，我上个周末在航班上，又速读了一半，其中看到他们的一个例子，如出一辙：

> 我们研究了一个通知 Basecamp 中的人群的想法。您不必一一勾选五个程序员，只需单击“程序员”，就会选择他们进行通知。当我们查看该产品时，我们看到了很多这种行为可能有意义的地方。如果我们让您在发布消息时选择一个群组，为什么在分配待办事项或在聊天室中提及人员时不可以呢？

> 我们决定该项目的核心价值是缩小消息通知对象的范围。我们明确地将其他情况标记为该项目的“越界”，并专注于我们想要的胜利：更快的发布消息流程。

Shape Up 里的方法是：

1. Setting the appetite：判断这件事值得花费多少时间和资源。
2. No gos：不做的事。也就是确定边界在哪里，什么事我们先不理。

当然，这里面另外有一个问题：我们提倡“举一反三”，提倡“想清楚再动手”，那么，这种粗糙，会带来“不思考就立即行动吗”吗？会不会带来更多“做了一半，才发现这事没想到”？

我希望不会，我希望产品同事：

1. 能想到一个需求背后会引出的问题，也能够举一反三地思考到位。
2. 能够恰当地确定边界，并不是想到了，就得实现它，而是告诉大家，还有哪些细节，但是我们目前先不实现。

所以这里是：想清楚，确定边界（什么不做）。

## 从中间开始

30% 的工作，70% 的效果，找到最佳平衡点

编剧写剧本，不管你能请到谁，想着最好的那个，能不能请到不重要，把最好的元素注入进去。

设想最理想的情况，如果这都不成立，就不成立了。

讨论产品交互时，Tony 经常提到一个观点：先做最有表现力，最能验证结论的部分。

- 往往这部分做完，上手体验，就知道事情是不是靠谱，是不是需要完全做出来了。
- 20% 的精力，实现 80% 的效果。或者 20% 的精力，就验证了事情是不是值得 100% 做完。我们做事，需要找到这个平衡点。
- 先做出不一样的、有挑战的、有创造力的部分后，再做“平凡”的部分，会觉得，平凡枯燥的体力活，也是有意义的。
- 有时候，做 demo 需要客户端和服务端配合，用“假协议”，快速粗糙地达到可点击、能上手体验，比精细化完整实现更有意义。
- 这种思想，比 MVP 还要再粗糙一点，能更快 Build 出有体感的产品。

在这种思想下，所谓画龙点睛，或许不应该把龙都画好了，再画眼睛，而是应该先做各种实验，把眼睛的不同可能性和表现形式探索出来，搞定了有表现力的眼睛，再画大家都能画好且大工作量的身体，会更高效。

这种思想，在阅读 Shape Up 的时候，也得到了同行印证，他们有个说法叫“从中间开始”——Start in the middle，他们提倡的也是不要按部就班，自下而上地垒砖块，而是抛开了其他一切问题，直接跳到中间有趣的部分。在他们的经验里，从中间开始，这里的“中间”的选择标准是：

1. 必须是核心。没有它，其它工作就毫无意义。
2. 必须小。小到可以在几天内可以验证完毕，否则就没有单独抽出来的意义。
3. 必须有趣。是以前没做过的、可以消除不确定性、可以增强团队信心的。

学会从中间开始，能极大提高效率——尤其是创业团队。大公司或许有他们的规范（大公司病），但创业团队不这样，就可惜了。
