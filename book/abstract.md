# 抽象

> 道生一，一生二，二生三，三生万物。
>
> —— 老子

> Write programs that do one thing and do it well. Write programs to work together.
> 写程序，让每个程序只做好一件事。写程序，让程序能协同工作。
>
> —— Doug McIlroy，UNIX 哲学

产品做着做着，会遇到一个时刻：表面上的定位撑不住了，需要重新想自己到底在做什么。

这时候有用的办法是往下看一层。不要盯着"语音笔记"或者"稍后阅读"这些场景标签，把它拆开，找到构成这个场景的原子能力。语音笔记拆开是输入、加工、输出三个环节。稍后阅读拆开是抓取、解析、筛选、摘要、推送、翻译六个能力。

拆到原子层，原来的场景消失了，新的组合浮现了。同一组积木，搭出完全不同的产品。

这就是我说的抽象：看穿产品的表面场景，找到底层的原子能力，然后用新的方式组合它们。在创业的语境下，它意味着不在场景层面思考竞争，往下挖一层，在原子能力层找机会。

知识星球相对稳定后，我抽出部分精力和一支出海团队合作，做了 Slax 系列产品，面向全球用户。做知识星球是回头看清楚了路。做 Slax 系列是正在路上，还不知道前面是什么。抽象这个思维方式，就是在做 Slax Note 和 Slax Reader 的过程中反复用到的。

## Slax Note：从语音笔记到内容原点

Slax Note 来自一个具体的困扰。我的灵感经常来自洗澡和散步，但这两个场景下掏手机打字很不方便。试过简略记几个词，几天后回顾常常忘了当时想的是什么。灵感就这样溜走了。

所以想做个工具：用语音快速记录，AI 帮我整理成可用的文字，方便复制到其他地方使用。

我知道这会是个很卷的市场。Evernote、Notion 都在叠加语音和 AI 能力，新笔记产品觉得是弯道超车的机会，操作系统内置的笔记在增强，各种会议记录产品、模型公司的 ChatBot 也在侵蚀。

调研了一圈同类产品。有的功能非常全面，改写级别、写作风格、输出长度、专业词汇、第三方集成，应有尽有。有的做得很漂亮，但往"问问 AI"、"待办"等方向走，想做的事情太多。还有大而全的，App、小程序、网页端都有。看完一圈，我反而看到了机会。大家都在做多、做全，我可以做少、做简单。

这不是为了偷懒，而是基于一个判断：用户需要的不是更多功能，而是更确定的体验。打开就知道能干什么，用完就能得到想要的结果。

Slax Note 的不一样，围绕一件事展开：灵感有输入、加工、输出三个环节，沿着这条线，一步步往深处走。

输入尽可能简单，拿起就用。录音、输入、记录灵感，只解决这一个问题。不做日记，不做待办，不做会议纪要。打开 App，说几句话，灵感就留住了。

加工由浅入深。最简单的是 Prompt，用户选一个 Prompt，AI 把语音转写的内容润色成不同风格的文字。再往上是 Skill，把多个步骤组合成一个技能，比如"把一段灵感变成小红书帖子+朋友圈文案+知识星球主题帖"。

沉淀是长期价值。做一个社区，让用户的 Prompt 和 Skill 可以相互交流。大家看到彼此是怎么使用这个产品的，学习和使用别人的技巧。绝大多数人不知道怎么写好一个 Prompt，但如果有个地方可以看到别人的玩法，一键套用，互相启发，社区就会慢慢长出价值。功能可以抄，社区和积累难抄。
产品上线后，我自己成了重度用户。写公众号的流程变了。以前坐下来打字、改，一篇文章要花 1 个小时。现在打开 Slax Note，看看记录过哪些灵感，挑一个，重听录音，脑子里梳理一下，对着手机直接说。10 分钟搞定。灵感不丢了，甚至感觉说话能力也在变好——录音多了，边想边说，语言组织能力在提升。

### 被挤压

但两个趋势正在侵蚀 Slax Note 的核心价值。

一方面，AI 输入法。豆包、微信输入法、Typeless 等产品把语音转文字变成了基础设施，铺到了所有应用里。另一方面，AI assistant。ChatGPT、OpenClaw 能直接接住语音输入并即时处理，比"先记后处理"更高效。

Slax Note 最初的核心是"语音转文字 + AI 润色"。当这层能力成为基础设施，产品的立足点就动摇了。剩余的核心场景变窄了：想起一件事，需要长篇大论地说，但当下不想深入处理。真实，但不够宽。

这就是夹缝生存的真实面貌。你找到了一条缝，开始生长，环境在变，缝隙在收窄。

### 拆开看

怎么办？

这时候抽象就派上用场了。把 Slax Note 从"语音笔记工具"这个场景标签里拆出来，重新审视输入、加工、输出这三个环节。

输入这一步，AI 输入法在吃。加工这一步，AI assistant 在吃。单看每一步，确实被挤得没空间了。

但整条链没有被任何单个工具吃掉。一段语音进来，后续不管经过多少加工——AI 展开、生成小红书帖子、发推特——所有衍生内容都挂在这条原始记录下面，形成一棵树。单步在贬值，链条在升值。

Slax Note 不是语音笔记工具，是"内容原点"。跟 Notion、Obsidian 的区别是：它们存的是结果，Slax Note 存的是从一句话到所有产出的完整链路。

具体做法是给 Slax Note 做 API 接口，让外部 AI 工具能调用。语音记到 Slax Note，外部工具通过 API 调出来处理，结果存回 Slax Note，长期沉淀。先不在内部建复杂的 skills 体系，而是通过 API 让外部工具处理，自己深度体验后再决定是否反哺回产品。

### 四个夹缝

拆到原子层之后，还看到了四个 AI assistant 和输入法难以替代的位置。

第一，私密思考空间。AI assistant 天然是对话式的，会回应你，有互动压力。Slax Note 可以做"不回嘴的容器"。AI 在后面默默结构化、打标签、关联，但不主动说话。日记、情绪、半成形的想法，需要安静的地方存着。

第二，移动端碎片捕捉。拼的是入口和摩擦力。iOS widget 一按就录、锁屏直接说、Apple Watch 抬腕就记。三秒钟的入口优势是 Telegram bot 做不到的，是原生 app 的结构性优势。

第三，多模态笔记。语音加图片加标注混合输入。看到东西拍张照加两句备注、白板草图配语音说明。这种混合输入在笔记 app 里很自然，在 AI assistant 的对话框里很别扭。

第四，内容发布上游。一段语音进来，自动生成多版本——推特、小红书、公众号——一键分发。从笔记工具变成个人内容 CMS。

这四个方向可以叠加，每加一层被替代的难度就高一层。单独一个功能容易被抄，但"极低摩擦的多模态输入 + 安静的沉淀空间 + AI 处理后回写 + 内容分发"这个组合很难复制。

一位同事跟我聊，他说自己没有语音笔记的习惯，对 Slax Note 没什么需求。但他说，如果能变成他的"智囊团"，他可能会用。想到一个主意，语音记录下来，用不同方式激发灵感，对一个观点做发散思考。这个反馈让我更确定了方向：加工能力是 Slax Note 的第二个战场。

这些方向能不能走通，写这本书的时候还不知道。但拆开看之后，至少知道了可以往哪里走。比被挤压时的焦虑好多了。

## Slax Reader：从书架到编辑部

2023 年 ChatGPT 火了，大家都在做 AI 产品。我却和一帮小伙伴在做看起来很陈旧的稍后阅读工具。很多朋友不理解。

但这可能是个小机会。

- 问题：我自己天天遇到。微信收藏夹、Safari 阅读列表、备忘录，散落在各处的链接和想法，找起来很痛苦。稍后阅读产品迭代慢，甚至有些动不动就不运营了。不是臆想的需求，是真实的痛点。
- 洞察：市场很小，现有团队迭代意愿不强，但有一小撮用户非常需要。AI 来了，有机会重构这个小东西，做出更好的伴读、推荐、速读、整理、输出工具。被大公司忽略，但对一小群人很重要。
- 实现：我们有所需的软件开发能力，AI 技术在文本、图片、视频理解上也相当成熟了。可以做出来。
- 土壤：这一项我还不确定。我自己是用户，代入思考，觉得我应该需要。但用户是不是真的接受，还在实验中。等你看到这本书的时候，或许就有些端倪了。

阅读是个小众市场，没什么人抢。别人看得懂但不屑做，觉得辛苦不挣钱。但对小团队来说，恰恰是最好的。

我甚至半开玩笑地跟朋友说：长期不怎么挣钱，也是护城河。大公司不会来抢，你可以安心做十年。就算商业上不划算，我刷自己的卡养着都愿意，毕竟我自己要用。

### 从书架到积木

Slax Reader 最初的模式是用户主动存 URL，被动阅读。一个书架。

同样的思路：把"阅读"这件事从场景里拆出来，往下看一层，可以分解成六个原子能力——抓取、解析、筛选、摘要、推送、翻译。

有意思的是，这六块积木的不同组合，覆盖完全不同的场景：

- 抓取（Hacker News RSS）→ 筛选（AI 相关）→ 摘要 → 推送
- 抓取（YouTube 频道）→ 解析（视频转文字）→ 摘要 → 翻译（英转中）→ 推送
- 抓取（SEC 财报页）→ 解析（PDF 提取）→ 摘要（关键指标）→ 推送

5 个积木能搭出 50 种用法。Slax Reader 从"书架"变成了"订阅报亭 + 私人编辑部"。

高手搭好的 pipeline 可以一键分享，普通用户直接用。跟 IFTTT 的 Applet、iOS 快捷指令一个逻辑：平台提供原子能力，社区提供组合方案。社区贡献的 pipeline 越多，平台越有价值。这也是长期护城河。

## 先有场景，再抽象成积木

抽象有个陷阱：先设计积木再做场景。

这很诱人。看到了原子能力，恨不得马上做一个完美的编排系统，让用户自由组合。但这是过度设计。做完几个具体场景后，才能看清哪些能力反复出现，积木的边界该怎么切。先设计积木再做场景，容易切错。

正确的顺序是反过来。先做具体场景——YouTube 摘要、Podcast 转文字、RSS 聚合——用户扔链接进来就能用。用户不关心底层架构，关心的是"扔一个 YouTube 链接进来，30 秒告诉我值不值得看"。

Slax Reader 的做法是：内部代码早点按积木思路写，但面向用户的界面先给预制场景。等场景够多了，开放自定义只是把内部能力暴露出来。两边不耽误。

Notion 也是这个路径。它把文档、表格、看板、数据库拆成了统一的 block 系统，用户用同一套积木搭出完全不同的工具。但 Notion 也是先有具体场景——文档和 Wiki——再逐步开放 block 的组合能力。

先有场景，再抽象成积木。不能反过来。

---

抽象跟黑客思维一脉相承：看穿系统的表面，找到底层的可操作单元，用意想不到的方式组合。

Slax Note 被挤压时，如果只在"语音笔记"这个场景里想办法，越想越窄。拆到原子层，反而看到了四个新方向。Slax Reader 如果只做"稍后读"，天花板很低。拆成六块积木，用户自己能组合出我们想不到的用法。

大公司在场景层竞争，小团队往下挖一层，在积木层找机会。往下看一层，路就宽了。
