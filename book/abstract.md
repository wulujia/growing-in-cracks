# 抽象

> 道生一，一生二，二生三，三生万物。
>
> —— 老子

> Write programs that do one thing and do it well. Write programs to work together.
> 写程序，让每个程序只做好一件事。写程序，让程序能协同工作。
>
> —— Doug McIlroy，UNIX 哲学

产品做着做着，会遇到一个时刻：曾经的定位不起作用，需要重新想自己到底在做什么。

这时候有一种办法是往下看一层。不要盯着产品当前的场景标签，把它拆开，找到底层的原子能力。拆到原子层，原来的场景还在，还能解决原本的问题，新的组合也浮现了。同一组积木，有机会搭出不同的产品，解决更丰富的问题。

这就是我这里想表达的抽象：从场景出发，向下拆解更基础的能力，用新的方式组合它们。

知识星球相对稳定后，我抽出部分精力和一支出海团队合作，做了 Slax 系列产品，面向全球用户。做知识星球是回头看清楚了路。做 Slax 系列是正在路上，还不知道前面是什么。下面用两个真实案例，讲讲抽象这个方法可以怎么用。

## Slax Note：从语音笔记到内容原点

Slax Note 来自一个具体的困扰。我的灵感经常来自散步，但这散步时下掏手机打字很不方便。试过简略记几个词，几天后回顾常常忘了当时想的是什么。灵感就这样溜走了。

所以我想做个工具：用语音快速记录，AI 帮我润色整理成可用的文字，方便复制到其他地方使用。

我知道，笔记是个很卷的市场。Evernote、Notion 都在叠加语音和 AI 能力，新笔记产品觉得是弯道超车的机会，操作系统内置的笔记在增强，各种会议记录产品、模型公司的 ChatBot 也在侵蚀。

调研了一圈同类产品。有的功能非常全面，改写级别、写作风格、输出长度、专业词汇、第三方集成，应有尽有。有的做得很漂亮，但往“问问 AI”、“待办”等方向走，想做的事情太多。还有大而全的，App、小程序、网页端都有。看完一圈，我反而看到了机会。大家都在做多、做全，我可以做少、做简单。

我们早期设想中，Slax Note 的不一样，围绕一件事展开：灵感有输入、加工、输出三个环节，沿着这条线，一步步往深处走。

- 输入尽可能简单，拿起就用。录音、输入、记录灵感，只解决这一个问题。不做日记，不做待办，不做会议纪要。打开 App，说几句话，灵感就留住了。
- 加工由浅入深。最简单的是 Prompt，用户选一个 Prompt，AI 把语音转写的内容润色成不同风格的文字。再往上是 skills，把多个步骤组合成一个技能，一次处理，多种结果。
- 输出去往多个地方。一段灵感加工完，变成小红书帖子、朋友圈文案、知识星球主题帖，复制就能用。我们做了社区，让 Prompt 和 skills 可以分享交流。大多数人不知道怎么写好 Prompt，但看到别人的玩法，一键套用，互相启发，社区慢慢长出价值。功能可以抄，社区和积累难抄。

产品上线后，我自己成了重度用户。写公众号的流程变了。以前坐下来打字、改，一篇文章要花 1 个小时。现在打开 Slax Note，看看记录过哪些灵感，挑一个，重听录音，脑子里梳理一下，对着手机直接说。10 分钟搞定。灵感不丢了，甚至感觉说话能力也在变好——录音多了，边想边说，语言组织能力在提升。

### 看对了，没做对

回头看，有两次我们看到了方向，但没能把事情做对。

GPT 刚出来没多久，我就意识到 AI 可以帮语音做润色。当时就提出做一个笔记产品，让用户录完音，AI 自动润色成可用的文字。方向看对了，切入点选错了。

真正把“AI 润色语音”这件事做成的，是输入法。豆包输入法、Typeless 这些产品，把语音转文字加润色做进了输入法，用户在任何 App 里都能用。这个能力被大众认可了，出圈了，传播了。而单独的笔记产品做润色，始终没能走进大众视野。

换个角度想，只有当输入法把这个能力做进去，大众才会意识到“原来语音可以这样用”。我们很早看见了需求，但选了笔记作为入口。Typeless 选了输入法，更自然地进入了用户心智。

第二次是 skills。我们很早就意识到，让用户输入一段话，用 AI 做各种处理——改写风格、生成帖子、提炼要点——这是一个很有特色的地方。当时还没有 skills 这个概念，但我们做的其实就是后来各家产品里的 skills。只是当时完全没有做服务端和存储，skills 只能保存在用户本地，没法云同步，重装系统就丢了。动作太慢，等我们还没做出来，Claude 已经把 skills 推出来了，各家 Chatbot 和 Agent 产品也都上了相应的功能。

### 被挤压

危险信号不是来自数据，是来自我自己的行为变化。

2026 年 2 月，我发现日常工作中我已经不用 Slax Note 了。要写一段文字，打开语音输入法直接说，说完就是可用的文字。要处理一个想法，打开 OpenClaw 直接对话，反复讨论，讨论完它还能帮我执行。“先记后处理”的流程被跳过了，因为记、想、做合成了一步。

产品的创造者不用自己的产品，这是最危险的信号。如果连我都觉得语音输入法加 AI Agent 更顺手，用户凭什么留在 Slax Note 里。

这就是夹缝生存的真实面貌。你找到了一条缝，开始生长，环境在变，缝隙在收窄。

### 拆开看

怎么办？

这时候抽象就派上用场了。把 Slax Note 从”语音笔记工具”这个场景标签里拆出来，重新审视输入、加工、输出三个环节。单看每一步，确实都在被吃。但三步连在一起呢？

有天晚上我在 Slax Note 里记了一条想法：”要不要给公司瘦身”。

记完之后想深入想想。但我不得不离开 Slax Note，打开 Telegram 找到 OpenClaw，把上下文重新说一遍。讨论了十几分钟，得到结论，让 AI 帮我发了封邮件。

流程断了两次。一次是离开 Slax Note，一次是重新交代上下文。

我想要的流程是：看到这条笔记，按住说话，直接围绕它展开讨论。AI 读取笔记内容作为上下文，不需要重新说一遍。讨论完，结论追加到这条笔记下面。说一句”整理成邮件发给我”，直接发。全程不离开 Slax Note。

如果流程跑通，笔记下面会挂着讨论记录、方案推演、发出去的邮件。一句模糊的想法，长成一个完整的决策。所有过程都在这棵树上。

这就是整条链的价值。输入法帮你把话说清楚，AI Agent 帮你把事想明白，但它们不存过程。Slax Note 存的是一个想法从萌芽到落地的完整链路。跟 Notion、Obsidian 的区别也在这里——它们存结果，Slax Note 存过程。不是语音笔记工具，是内容原点。单步在贬值，链条在升值。

输入法丢了一样东西：语音转成文字，原始录音就没了。Slax Note 留着原始语音。语气、停顿、犹豫，文字抹掉的东西，录音里都在。这些录音日积月累，是其他工具里不会有的资产。

AI Agent 也有一个够不到的位置。目前的 AI Agent，不管是 ChatGPT、Claude 还是 Telegram 里的 bot，都是聊天窗。对话像流水一样往下走，聊完就过去了。Slax Note 更像论坛——围绕一个主题不断沟通、修改、深化。有些想法不想要 AI 回应，需要一个安静的地方存着，AI 在后面默默结构化，但不主动说话。另一些想法需要围绕主题深入讨论。聊天窗是流水，论坛是沉淀——知识星球的与 ChatBot 的不一样，和这也有点像。

一位同事跟我聊，他说自己没有语音笔记的习惯，对 Slax Note 没什么需求。但他说，如果能变成他的”智囊团”，他可能会用——想到一个主意，语音记录下来，用不同方式激发灵感，做发散思考。这跟那个晚上的体验指向同一件事：加工能力是关键。

顺便提一句：我在开始测试的时候，是请同事把 Slax Note 的能力快速包装成了 skills，让 OpenClaw 学习调用，然后尝试在 OpenClaw 里讨论、直接输出到 Slax Note。一方面是最小化的测试，另一方面也是 MVP，最小化地找体感。

Slax Note 卡在一个小小的夹缝里。这个夹缝跟知识星球当年在微信生态里找到的缝隙很像。当我意识到这一点的时候，觉得 Slax Note 有了它自己的生长空间。

这个方向能不能走通，写这本书的时候还不知道。但拆开看之后，知道了还可以往哪里走。比被挤压时的焦虑好多了。

## Slax Reader：从书架到编辑部

2023 年 ChatGPT 火了，大家都在做 AI 产品。我却和一帮小伙伴在做看起来很陈旧的稍后阅读工具。很多朋友不理解。

但这可能是个小机会。

- 问题：我自己天天遇到。微信收藏夹、Safari 阅读列表、备忘录，散落在各处的链接和想法，找起来很痛苦。稍后阅读产品迭代慢，甚至有些动不动就不运营了。不是臆想的需求，是真实的痛点。
- 洞察：市场很小，现有团队迭代意愿不强，但有一小撮用户非常需要。AI 来了，有机会重构这个小东西，做出更好的伴读、推荐、速读、整理、输出工具。被大公司忽略，但对一小群人很重要。
- 实现：我们有所需的软件开发能力，AI 技术在文本、图片、视频理解上也相当成熟了。可以做出来。
- 土壤：这一项我还不确定。我自己是用户，代入思考，觉得我应该需要。但用户是不是真的接受，还在实验中。等你看到这本书的时候，或许就有些端倪了。

阅读是个小众市场，没什么人抢。别人看得懂但不屑做，觉得辛苦不挣钱。但对小团队来说，恰恰是最好的。

我甚至半开玩笑地跟朋友说：长期不怎么挣钱，也是护城河。大公司不会来抢，你可以安心做十年。就算商业上不划算，我刷自己的卡养着都愿意，毕竟我自己要用。

### 不一样在哪

起点很简单：收藏不丢。收藏链接时自动缓存全文。链接 404、作者删帖、网站倒闭，都不影响。这些在竞品里是付费功能，我们免费提供。用了二十年稍后阅读工具，丢链接一直是最大的困扰。从这个痛点出发。

AI 放在核心位置。不是在书签工具上加个 AI 按钮，而是反过来——做一个 AI 阅读助手，顺便帮你管书签。收藏一篇长文，点一下，30 秒出摘要和大纲，判断值不值得读。读的时候遇到问题，直接问内置的 AI，不用切窗口。以前我阅读时总要另开一个浏览器窗口放着 ChatGPT，随时唠两句。现在不用了。

社交阅读。这个方向几乎没有竞品在做。划线、评论可以分享，其他人能在分享页讨论。读同一篇文章的人，看到彼此的批注和提问。

开源。2024 年，Omnivore 被收购后关停，用户被要求一个月内导出数据。之前 Pocket 被 Mozilla 收购后也逐渐边缘化。稍后阅读工具关停，用户积累的收藏、划线、笔记跟着消失。开源是我们的回答：代码在 GitHub 上，Apache 2.0 许可，支持自托管。就算有一天我们不做了，产品不会消失。

还有一件事在摸索：怎么解决“收藏了永远不读”。以前我的 Read it later 列表只增不减，最后变成了 Never Read It。AI 解析帮了一部分——快速判断值不值得读，不值得的直接归档，列表终于可以 Inbox Zero。但更深层的问题是怎么让用户形成阅读习惯，类似 Duolingo 催人学习的那套机制。我们还有蛮多想法，团队里喜欢阅读的同事们还在实验。

### 往哪走

工具初步做好之后，发出去，不温不火。用户不多，数据始终不太好。

大家开始犹豫应该往哪个方向突破。

一个方向是往重了做。社交阅读，让所有 URL 都能在 Slax Reader 里讨论，成为全世界 URL 的社区——类似 Goodreads 对书的作用，但面向文章和网页。用户订阅，把看到的好内容通过付费或免费的形式分享。这些想法都好，但离现实还远。社交阅读要有人用才能转起来，现在用户基数还不够。只能一点一点搭积木，先把基础功能做扎实。

另一个方向来自用户的声音。不断有人问：能不能支持 PDF？能不能支持 RSS？YouTube 视频能不能放进来？B 站呢？Podcast 呢？

这些需求看起来零散，但拼在一起，指向同一件事：用户想往 Slax Reader 里放的，不只是网页链接。他们想把所有值得读、值得看、值得听的东西都收进来。我们还有一个愿景：内容收进来了以后，大家可以在这里共同讨论。

这就是从“书架”走向“积木”的起点。

### 从书架到积木

以下是我最新的思考，还没有完全实现。

Slax Reader 最初的模式是用户主动存 URL，被动阅读。一个书架。

同样的抽象方法：把”阅读”这件事从场景里拆出来，往下看一层，可以分解成六个原子能力——抓取、解析、筛选、摘要、推送、翻译。

有意思的是，这六块积木的不同组合，覆盖完全不同的场景。同样是"帮我盯着一个信息源，有更新就处理好推给我"，因为信息源不同、处理方式不同，需要的积木组合也不同：

- 抓取（Hacker News RSS）→ 筛选（AI 相关）→ 摘要 → 推送
- 抓取（YouTube 频道）→ 解析（视频转文字）→ 摘要 → 翻译（英转中）→ 推送
- 抓取（SEC 财报页）→ 解析（PDF 提取）→ 摘要（关键指标）→ 推送

如果这件事情能够跑通，6 个积木能搭出 50 种用法。Slax Reader 从“书架”变成了“订阅报亭 + 私人编辑部”。

高手搭好的 pipeline 可以一键分享，普通用户直接用。跟 IFTTT 的 Applet、iOS 快捷指令一个逻辑：我们提供原子能力，社区提供组合方案。社区贡献的 pipeline 越多，产品越有价值。这也是长期护城河。

## 先有场景，再抽象成积木

抽象有个陷阱：先设计积木再做场景。

这很诱人。看到了原子能力，恨不得马上做一个完美的编排系统，让用户自由组合。但这是过度设计。做完几个具体场景后，才能看清哪些能力反复出现，积木的边界该怎么切。先设计积木再做场景，容易切错。

正确的顺序是反过来。先做具体场景——YouTube 摘要、Podcast 转文字、RSS 聚合——用户扔链接进来就能用。用户不关心底层架构，关心的是“扔一个 YouTube 链接进来，马上告诉我值不值得看”。

Slax Reader 应该是：内部代码早点按积木思路写，但面向用户的界面先给预制场景。等场景够多了，开放自定义只是把内部能力暴露出来。两边不耽误。

Notion 也是这个路径。它把文档、表格、看板、数据库拆成了统一的 block 系统，用户用同一套积木搭出完全不同的工具。但 Notion 也是先有具体场景——文档和 Wiki——再逐步开放 block 的组合能力。

先有场景，再抽象成积木。不能反过来。

回头看这两个案例。Slax Note 把语音笔记拆成输入、加工、输出三个环节，发现单步在贬值但链条在升值，从”语音笔记工具”变成了”内容原点”。

Slax Reader 把稍后阅读拆成抓取、解析、筛选、摘要、推送、翻译六个能力，发现不同组合覆盖完全不同的场景，从”书架”变成了”编辑部”。

方法是同一个：往下看一层，找到原子能力，用新的方式组合。

## 为什么写一个还没有结局的故事

写这一章的时候，我有点犹豫。Slax 系列还没有被验证。有人说，成功经验才有说服力，正在进行的实验不算数。万一失败了，写进书里不尴尬吗。

想了想，觉得不对。

这本书叫“夹缝生长”。夹缝生长不是一个结果，是一个过程。如果只写已经成功的故事，那不是夹缝生长，是成功学。

Slax Note 被语音输入法和 AI Agent 挤压，被迫重新思考定位。Slax Reader 在一个没人看好的小市场里摸索方向。这些挣扎是真实的。你读到这里的时候，这些产品可能已经跑通了，也可能不温不火，甚至可能已经关掉了。都没关系。

重要的不是结果，是这个过程：发现缝隙在收窄，往下看一层，找到新的可能性，然后带着假设去验证。这才是小团队的日常。我们没有办法一路高歌猛进，只能一步步在夹缝里往前拱。
