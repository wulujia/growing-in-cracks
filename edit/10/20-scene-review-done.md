# 《现场》审阅意见

> 全书第 20 章 / 共 40 章

## Q1: 共识与非共识

**共识部分：**
- "到现场去"本身就是常识。深入用户调研、做用户访谈、收集用户反馈、建立 NPS 机制，这些都是产品管理的标准动作。
- "Do things that Don't Scale"也是 YC 创业课的经典教条，已经被反复引用。
- "客户第一"、"一切以用户价值为依归"这些表述在互联网圈已经被说烂了。

**非共识的可能性：**

但这章最有价值的不是"要到现场"这个共识，而是几个被忽视的反共识观点：

1. **"我们是服务者，不是管理者"** —— 很多产品团队在用户群管理上其实是管理者心态，移除"捣乱用户"是常规操作。这章指出了这个问题，但展开不够。可以强化这个观点：社群不是你的权力场域，而是用户的服务空间。

2. **"客户第一不等于客户说什么就给什么"** —— 这个观点有价值，但被夹在生财有术的案例里，力度不够。应该更明确地指出：到现场不是去当客户的传声筒，而是去理解客户背后的真实需求。很多产品经理误把"用户反馈"当成"产品需求"，这是错的。

3. **"坐在办公室里想不出好产品"vs"在现场也可能被用户误导"** —— 这章只讲了前者，没讲后者。可以补充：到现场不是去照单全收，而是去验证假设。用户会撒谎，用户会不知道自己要什么，用户会提出伪需求。现场的价值是观察用户的行为，而不是记录用户的话语。

**建议补充的非共识观点：**

- **反共识1：好的反馈不是最多的反馈，而是最极端的反馈。** 阳志平的反馈价值高，不是因为他代表大多数用户，而是因为他是极端用户。极端用户的痛点，往往预示着未来大众用户的痛点。

- **反共识2：现场的价值不是收集需求，而是发现矛盾。** 真正有价值的现场观察，不是用户说"我要这个功能"，而是用户说"我要A"但行为却是"在做B"。这种矛盾才是产品创新的来源。

- **反共识3：规模化的现场才有意义。** 这章讲了很多"到现场"的案例，但没有讲清楚：一次两次的现场是没用的。必须建立持续的、规模化的现场机制，才能从噪音中提取信号。否则你到的"现场"可能只是个例。

**具体建议：**
- 在"为什么要在现场"这一节，增加一段：现场的陷阱——用户会骗你，数据也会骗你，只有持续观察行为才能接近真相。
- 在"从现场到产品"这一节，增加一个案例：某个用户强烈要求的功能，我们做了，但数据显示根本没人用。这才是真实的创业场景。

## Q2: 知识星球故事建议

这章已经有不少知识星球的故事，但都是碎片化的，缺乏完整性。可以补充以下类型的故事：

**1. 从现场观察到产品迭代的完整闭环案例（建议补充）**

- **故事主线：** 阳志平提出"仅自己可见"功能 → 我们内部讨论了什么 → 设计方案是什么 → 上线后数据如何 → 用户反馈如何 → 这个功能最终演变成了什么。
- **为什么重要：** 现在这个案例只讲了"阳老师提了需求"，但没讲"我们怎么做的"。读者会好奇：这个功能最后做了吗？怎么设计的？效果如何？

**2. 一个失败的现场案例（建议补充）**

- **故事主线：** 某次星主交流会，收集到了大量反馈，我们认真做了，但上线后发现用户根本不买账。我们反思了什么？
- **为什么重要：** 这章现在太"成功学"了，全是"到现场就能发现好需求"。但真实情况是：大部分现场反馈都是噪音，只有少数是信号。需要一个失败案例来平衡。

**3. 一个"沉默用户"的故事（建议补充）**

- **故事主线：** 一个长期使用但从不反馈的用户，有一天突然流失了。我们通过什么方式找到他？他说了什么？我们改变了什么？
- **为什么重要：** 这章讲了很多"主动找用户"和"让用户找到你"，但没讲"那些不说话的用户怎么办"。NPS 那段提到了这个问题，但没有具体案例。

**4. 一个"反直觉"的用户行为观察（建议补充）**

- **故事主线：** 某个功能，星主们强烈反对，但我们通过数据发现普通用户很喜欢。我们怎么处理这个矛盾的？
- **为什么重要：** 星主和普通用户的需求不一致，这是知识星球的结构性矛盾。这个矛盾在"现场"这一章应该被展开。

**5. 批量购买邀请链接的完整故事（建议展开）**

- **现状：** 现在只讲了"星主反馈问题 → 我们开会解决"，但没讲清楚：
  - 这个问题是怎么被发现的？是一个星主提的，还是很多星主都遇到了？
  - 我们的解决方案具体是什么？
  - 上线后效果如何？
  - 有没有产生新的问题？
- **为什么重要：** 这是从现场到产品的经典案例，但现在写得太简略，读者看不到完整的决策过程。

## Q4: 整体评分

**评分：72/100**

**扣分明细：**

**1. 结构松散，缺乏核心论点（-10分）**

这章的问题不是内容不好，而是缺乏一个清晰的核心论点。

读完全章，我能总结出的主要观点是：
- 要到现场去
- 到现场可以发现真实需求
- 到现场会遇到各种问题
- 从现场可以改进产品

但这些观点都很表面。真正的核心论点应该是：**现场不是用来收集需求的，而是用来验证假设、发现矛盾、理解用户行为的。**

建议：
- 在开头明确提出核心论点
- 全章围绕这个论点展开
- 结尾回扣这个论点并升华

**2. 案例质量参差不齐（-8分）**

好的案例：
- 阳志平的"仅自己可见"反馈（有场景、有洞察）
- 古典老师的5个问题（有方法、可复用）
- 珍大户的内容策略（有深度、有启发）

差的案例：
- 批量购买邀请链接的故事（讲了一半就停了，没讲清楚）
- 错误推送"内容违规"消息的故事（只讲了"我们犯错了"，没讲"我们学到了什么"）
- 生财有术减免手续费的故事（这个故事更适合放在"价值观"那章，而不是"现场"这章）

建议：
- 每个案例都要有完整的"发现问题 → 分析问题 → 解决问题 → 反思收获"结构
- 删掉那些只有"发现问题"没有"解决方案"的案例

**3. 缺乏方法论沉淀（-5分）**

这章讲了很多"到现场"的故事，但没有提炼出可复用的方法论。

比如：
- 如何识别哪些用户反馈是真需求，哪些是伪需求？
- 如何在100条反馈中找到最重要的那一条？
- 如何建立一个可持续的"现场机制"，而不是靠创始人亲自跑？

建议：
- 增加一节"现场方法论"，提炼3-5个可操作的框架
- 比如："判断需求真伪的3个问题"、"优先级排序的2x2矩阵"

**4. 与全书框架的关联不够强（-3分）**

这本书的核心是"快与慢、大与小、繁与简"三大矛盾，以及"分形思维"和"黑客思维"。

但这章几乎没有提到这些。"现场"这个概念可以和这些主题关联：
- **快与慢：** 到现场是"慢"的功夫，但能换来产品迭代的"快"
- **大与小：** 大公司靠数据决策，小公司靠现场洞察
- **黑客思维：** 到现场不是按部就班做调研，而是用"黑客"的方式快速试错

建议：
- 在开头或结尾，明确点出"现场"在全书框架中的位置
- 至少有2-3处呼应全书的核心思想

**5. 语言风格不够简洁（-2分）**

这章有些段落太长，不符合"每段最多8句话"的要求。

比如第23-26行是一个段落，但只有4句话，符合要求。
但第44-54行是一个段落，有11句话，太长了。

建议：
- 重新断句，确保每段不超过8句话
- 删掉一些不必要的形容词和副词

**6. 缺乏金句和升华（-2分）**

这章的结尾是："好产品不是在办公室里想出来的，是在现场一点一点长出来的。"

这句话还可以，但不够有力。和前文的"坐在办公室里，想不出好产品"重复了。

建议：
- 结尾不要重复开头的观点，而是要升华
- 比如："现场不是用来收集答案的，而是用来发现问题的。好产品不是满足用户需求，而是解决用户矛盾。"

## Q5: 必要性审视

**不应该砍掉这一章。**

理由：

**1. 这一章填补了重要空白**

全书第三部分"具体执行"分为三个子部分：
- 做产品（第11-16章）
- 做增长（第17-23章）
- 调方向（第24-29章）

在"做增长"这个子部分中：
- 第17章《手把手》讲的是用户教育
- 第18章《收入》讲的是商业化
- 第19章《内容输出》讲的是内容营销
- 第20章《现场》讲的是用户调研
- 第21章《市场推广》讲的是获客
- 第22章《品牌》讲的是长期建设

如果砍掉《现场》，就缺少了"用户调研与反馈"这个关键环节。产品迭代的闭环就断了。

**2. 这一章的核心价值被低估了**

虽然这章现在的完成度只有72分，但它的核心价值很高：**在一个充斥着增长黑客、数据驱动、AI决策的时代，提醒创业者不要忘记"到现场去"这个最原始但最有效的方法。**

这是一个反共识的观点。

大部分创业者都在追求"规模化"、"自动化"，但小团队的优势恰恰是"不规模化"。你可以亲自去见用户，这是大公司做不到的。

**3. 这一章的案例资源丰富**

知识星球的"星主交流会"、"用户吐槽会"、"NPS调研"，这些都是非常具体的实践案例。只要把这些案例讲透，这章的价值会非常高。

**但需要大幅改进：**

如果这章维持现在的质量出版，会拉低全书的水平。但如果按照上面的建议改进，这章可以成为全书的亮点之一。

**改进方向：**
1. 明确核心论点："现场不是用来收集需求的，而是用来验证假设、发现矛盾、理解用户行为的"
2. 补充非共识观点："用户会骗你"、"极端用户比平均用户更重要"、"规模化的现场才有意义"
3. 完善案例：每个案例都要有完整的闭环
4. 提炼方法论：给出3-5个可操作的框架
5. 关联全书主题：呼应"快与慢、大与小、繁与简"

**结论：不砍。但必须改。**
