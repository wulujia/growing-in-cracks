# 《计算子弹》审阅意见

> 全书第 29 章 / 共 40 章

## Q1: 共识与非共识

这一章讲的基本是常识。"计算现金流"、"粮草厚度"、"先子弹后炮弹"、"控制固定成本"——任何财务管理书籍都会讲这些。

真正的非共识应该在这里：

1. **"默认会死"状态下的正确策略**
   - 大部分创业指导都在讲如何增长，但对"默认会死"的公司该怎么办，讲得含糊。
   - 非共识：当你确认自己"默认会死"时，应该立即进入"战时状态"，而不是继续按照"和平时期"的打法执行。具体来说：
     - 立即砍掉所有非核心业务（即使它们有收入）
     - 把剩余子弹全部压在最有可能突破的单点上
     - 接受极端的不平衡状态：宁可一个点做到 90 分，也不要三个点各做到 60 分
   - 这违背了"多元化降低风险"的常识，但在子弹有限时，分散就是死亡。

2. **"粗糙的快"比"精确的慢"更危险**
   - 文中说"先快再准"，但没有指出一个反常识的陷阱：
   - 很多创业者误把"MVP"理解成"可以烂"，快速做出一堆没人要的东西，浪费了子弹还以为在"精益创业"。
   - 真正的非共识：快的前提是你已经通过深度思考缩小了探索范围。如果方向错了，快只会让你更快地死。
   - Slax Note 的例子恰恰证明了这一点——用 GPTs 验证不是"无脑快"，而是"想清楚了核心假设后的快"。

3. **子弹成本的"隐性浪费"**
   - 文中算出每次产品迭代成本 70-80 万，这是显性成本。
   - 非共识的隐性成本：
     - 机会成本：这 3 周本可以验证另一个方向
     - 认知成本：失败的尝试会让团队产生"我们已经试过了"的惯性
     - 时间成本：粮草厚度不只是钱的问题，也是市场窗口期的问题
   - 应该补充：每发子弹的真实成本 = 账面成本 × 2（因为你失去了用这些资源做别的事的机会）

**建议增加的非共识观点：**

- 在"默认会死"状态下，融资不是解药，是毒药（稀释股权、被迫增长、失去灵活性）。真正的出路是砍到极致，找到能自我造血的最小单元。
- 粮草厚度在 6-12 个月时，最应该做的不是"节流"或"开源"，而是"找杠杆"——找到投入 1 能产出 5 的那个点，all-in。
- 固定成本的真正危险不是金额大小，而是"心理锁定"——租了大办公室就会想招人填满它，招了人就会想给他们找事做，最后钱不是花在刀刃上，而是花在维持"公司样子"上。

## Q2: 知识星球故事建议

知识星球的故事应该是这章的核心支撑，但目前只在最后一段轻描淡写地提了一句。这是巨大的浪费。

**建议补充的知识星球故事：**

1. **2018 年的决策时刻（贯穿全章的主线）**
   - 当时账上 930 万，粮草厚度不到 12 个月。
   - 具体支出构成是什么？人力成本多少？服务器多少？
   - 摆在桌面上的选项有哪些？（裁员？融资？砍产品线？）
   - 最终选择 all-in 知识星球的决策过程是怎样的？
   - 这个决策在当时看来有多大风险？团队内部有没有反对意见？

2. **知识星球的"子弹成本"案例**
   - 知识星球某次关键功能迭代，投入了多少子弹？
   - 结果如何？是打中了（数据显著改善）还是打偏了（做了没人用）？
   - 如果打偏了，后续如何调整？损失了多少子弹？

3. **知识星球如何从"默认会死"变成"默认能活"**
   - 2018 年时，知识星球的增长曲线是怎样的？
   - 盈亏平衡点是什么时候达到的？
   - 哪个动作是"从 0 到 1"的关键转折点？是某个功能？某个运营策略？还是某个市场契机？
   - 这个过程花了多长时间？消耗了多少子弹？

4. **知识星球在成本控制上的具体做法**
   - 2018 年前后，知识星球团队规模是怎样的？
   - 如何做到"省着打"？哪些成本砍了？哪些成本保留了？
   - 有没有因为过度节省而踩坑的例子？

5. **知识星球某个"先子弹后炮弹"的案例**
   - 例如：付费功能、社区功能、内容工具等，哪个是先用小成本验证，再大力投入的？
   - 子弹阶段怎么打的？（最小可行版本是什么样的？）
   - 什么数据让你确认这个方向是对的？
   - 炮弹阶段投入了多少？结果如何？

**具体插入位置建议：**

- 第 37 节"粮草厚度"：插入 2018 年知识星球的真实财务状况，用具体数字展示粮草厚度的计算和危机感
- 第 82 节"子弹先于炮弹"：补充知识星球某个具体功能的"先子弹后炮弹"案例，替换或补充 Canva 案例
- 第 110 节"打得准还是打得快"：Slax Note 案例可以保留，但应该再补充知识星球早期快速试错的案例
- 第 140 节结尾：大幅扩写，详细讲述 2018 年的决策过程和后续如何逆转局面

目前这章最大的问题是：知识星球作为作者亲历的案例，信息密度反而不如 Canva 这种二手信息。这是不对的。

## Q3: 图片与表格建议

这章逻辑清晰，结构完整，但确实有几处可以用图表增强表达效果。

**建议 1：粮草厚度示意图（第 37-67 节）**
- 用柱状图或折线图展示：
  - X 轴：时间（月份）
  - Y 轴：现金余额
  - 不同颜色标注 12 个月、6 个月、3 个月的危险区间
  - 可以叠加一条支出曲线和收入曲线，直观展示"默认能活"和"默认会死"的区别
- 配上知识星球 2018-2019 年的真实数据，会非常有说服力

**建议 2：子弹与炮弹的成本对比表（第 82-109 节）**
- 表格：
  | 阶段 | 成本 | 时间 | 验证目标 | 结果 |
  |------|------|------|----------|------|
  | 子弹 1 | X 万 | X 周 | 验证需求是否存在 | 失败/成功 |
  | 子弹 2 | X 万 | X 周 | 验证价格敏感度 | 失败/成功 |
  | 炮弹 | XX 万 | X 月 | 全力推广 | 成功 |
- 用知识星球或你们团队其他产品的真实数据填充，比抽象讲"先子弹后炮弹"有力得多

**建议 3：固定成本 vs 可变成本对比（第 126-138 节）**
- 饼图或柱状图：
  - 展示 2018 年前后公司成本结构的变化
  - 固定成本（人力、租金、固定服务费）vs 可变成本（外包、按量付费服务、奖金）
  - 配上具体金额和占比
- 这会让"人是最大的固定成本"这个观点更具象

**建议 4：12-6 法则的决策树（第 59-65 节）**
- 流程图：
  ```
  计算粮草厚度
    ├─ >12 个月 → 相对从容，可准备融资
    ├─ 6-12 个月 → 立即启动融资 + 节流开源
    ├─ 3-6 个月 → 紧急决策：融资/砍成本/增收
    └─ <3 个月 → 最小化运行/体面关闭
  ```
- 配上每个阶段的具体行动清单

**建议 5：Canva 的"子弹到炮弹"时间线（第 98-106 节）**
- 时间轴图：
  - 2007: Fusion Books（子弹）
  - 2010-2012: 验证方向，准备 Canva
  - 2013: Canva 上线（炮弹）
  - 2014: 75 万用户
  - 2023: 1.25 亿用户
- 用图示展示"子弹验证 → 炮弹放大"的时间跨度和效果差异

**优先级排序：**
1. 粮草厚度示意图（最重要，最直观）
2. 固定成本 vs 可变成本对比（最实用）
3. 12-6 法则决策树（最易操作）
4. 子弹与炮弹的成本对比表（最有说服力，但需要详实数据）
5. Canva 时间线（次要，因为不是自己的案例）

## Q4: 整体评分

**评分：72/100**

这章的优点很明显：
- 结构清晰，逻辑完整（粮草厚度 → 子弹策略 → 快与准 → 成本控制 → 真实案例）
- 语言简洁，符合全书风格
- 核心概念"粮草厚度"、"先子弹后炮弹"、"默认能活 vs 默认会死"都很有价值
- Warren Buffett 和 Benjamin Franklin 的引言恰当
- 开头和结尾呼应，形成闭环

但扣掉 28 分，主要问题在：

**1. 缺乏独特洞察（-10 分）**
- 全章都在讲"常识"，没有提出反常识的观点
- "计算现金流"、"控制成本"、"快速试错"——这些任何财务管理或创业书都会讲
- 读完后感觉"有道理，但我好像早就知道"
- 缺少那种"原来还可以这样想"的惊喜感

**2. 自己的故事讲得太浅（-10 分）**
- 2018 年账上只剩 930 万是个绝佳的开头，但后续展开不足
- 最后一段匆匆结尾："收入超过了支出，知识星球活下来了"——然后呢？
  - 这个过程具体怎么做的？
  - 做了哪些决策？
  - 经历了哪些挣扎？
  - 有没有踩坑？
  - 团队内部有没有分歧？
- 读者会觉得：你就是讲了个故事的开头和结尾，中间最精彩的部分藏起来了
- Canva 的案例反而讲得比自己的案例详细，这是本末倒置

**3. 缺乏可操作的细节（-5 分）**
- "12-6 法则"提出了一个框架，但每个阶段具体应该做什么？
  - 6-12 个月时说"必须认真考虑节流和开源"——具体怎么节流？怎么开源？
  - 3-6 个月时说"考虑最小化运行"——最小化到什么程度？哪些能砍？哪些不能砍？
- "粮草厚度 = 现金余额 / 每月净支出"——这个公式太简化了
  - 很多公司的支出不是线性的（有季节性、有突发支出）
  - 应收账款怎么处理？（文中提到了"只算现金"，但如果有大额应收即将到账呢？）
  - 至少应该给出一个"保守算法"和"乐观算法"

**4. 数据支撑不足（-3 分）**
- "每次产品迭代成本 70-80 万"——这个算法过于粗糙
  - 怎么算出来的？能不能展示一下计算过程？
  - 读者无法判断这个数字的可信度
- 如果能给出知识星球 2018-2019 年的真实财务数据（哪怕是脱敏的），会大大增强说服力

**5. 前后呼应不够（-0 分，但有改进空间）**
- 开头引用了 Warren Buffett 和 Benjamin Franklin，但正文没有再提到"永远不要亏钱"和"不做准备就是准备失败"这两个核心思想
- 如果能在具体案例中呼应这两句话，会更有力

**优秀的地方（值得保留）：**
- "先子弹后炮弹"的比喻很生动，引用《选择卓越》恰当
- "默认能活 vs 默认会死"这个框架很有价值
- "只算现金，不算应收账款"——这个提醒很实用
- Slax Note 用 GPTs 快速验证的案例简洁有力

**改进后可以到 85-90 分的路径：**
- 补充反常识观点（+5 分）
- 大幅扩写知识星球的案例，讲清楚 2018 年的决策过程和执行细节（+8 分）
- 增加可操作的细节清单（+3 分）
- 补充数据支撑（+2 分）

## Q5: 必要性审视

不能砍。

理由：

1. **承上启下的结构价值**
   - 这是第四部分"产品之外"的开篇章
   - 前三部分讲"怎么做产品"，这章开始讲"怎么让产品活下去"
   - "计算子弹"是"钱"的问题的起点，后面还会讲"风险控制"、"投资"、"公司规模"
   - 如果砍掉这章，第四部分会失去一个明确的入口

2. **填补全书的关键空白**
   - 前三部分是"理想主义"的（怎么做好产品），第四部分是"现实主义"的（怎么活下去）
   - 很多创业书只讲前者，不讲后者，或者浅尝辄止
   - 这章虽然讲的是常识，但恰恰是很多创业者会忽略的"要命的常识"
   - 没有这章，全书会显得"不接地气"——光讲产品理念，不讲生存压力

3. **读者的实际需求**
   - 这本书的目标读者是"小团队创业者"和"独立开发者"
   - 他们最关心的问题之一就是："我还能撑多久？"
   - 如果全书只讲产品方法论，不讲生存策略，会让读者觉得"说得都对，但我活不到那一天"

4. **为后续章节铺垫**
   - 这章提出了"粮草厚度"、"默认能活 vs 默认会死"等概念
   - 后续章节（风险控制、投资、公司规模）都会用到这些概念
   - 如果砍掉这章，后面的章节会缺少铺垫

**但这章可以更好：**
- 目前的版本是"及格线以上"，但还没到"不可替代"
- 如果能补充更多知识星球的真实案例，加入反常识的洞察，这章会从"必要"变成"精彩"

**结论：**
不能砍，但必须改。目前的版本保证了结构完整性，但没有发挥出这个主题的真正价值。如果能把 2018 年的故事讲透，把"默认会死"状态下的决策细节写清楚，这章会成为全书最有生存感、最接地气的一章。
