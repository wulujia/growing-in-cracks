# 《粗糙》审阅意见

> 全书第 13 章 / 共 40 章

## Q1: 共识与非共识

这一章说的，大部分是常识。"先快速迭代再完善"、"MVP 思想"、"不要过早优化"，这些都是创业圈反复强调的共识。

但问题在于：共识不等于能做到。大多数人知道要快速发布，但实际做的时候还是拖延、纠结、追求完美。所以这一章的价值，不在于提出新观点,而在于用丰富的案例和具体做法,让人真正理解"粗糙"的边界在哪里。

**非共识的地方在哪里？**

1. **粗糙的边界在变化**：这一章的后半部分提到"现在的粗糙更难"，指出了一个重要的非共识——MVP 的门槛在提高。很多人还停留在"能用就行"的认知里，但现在用户的容忍度已经降低了。这个观点应该更突出，甚至可以作为全章的核心矛盾来展开。

2. **MEP vs MVP**：最小可进化产品（MEP）的概念是个好的非共识点。但现在只是放在最后一节，像个补充说明。应该把这个概念提前，和 MVP 对比着讲，让读者明白：不只是"能用"，还要"能进化"。

3. **粗糙的代价**：这一章几乎都在讲粗糙的好处，但没有充分展示粗糙的代价。比如：
   - 知识星球早期不考虑续费，后来怎么补的坑？
   - 当机三次之后，用户流失了多少？
   - Gumroad 手动给用户付款，规模到多大时这个方法就彻底不行了？

   讲清楚代价，读者才能真正理解"什么时候该粗糙，什么时候该完美"。

4. **粗糙不等于不思考**：这一章有一句"想清楚，但不是做完"，但展开不够。很多人会误解为"先做再说"，但其实应该是"深度思考 + 快速执行最小范围"。这个区别应该更明确。

**建议添加的非共识观点：**

- **粗糙是有选择的粗糙**：不是所有地方都粗糙，而是要识别哪些地方必须精致（关键体验），哪些地方可以粗糙（非关键功能）。这需要判断力。
- **粗糙需要设计**：实验室功能、从中间开始、用时间框约束规格，这些都是"粗糙的方法论"。粗糙不是随意，而是有意识的策略。
- **粗糙是为了对话**：快速发布的目的不只是节省时间，而是尽早和真实用户对话，获得反馈。这是一个从"闭门造车"到"开放共建"的思维转变。

## Q2: 知识星球故事建议

这一章已经有知识星球的案例（不考虑续费、并发当机、快捷键功能），但都比较简短，缺乏具体的"痛感"和"转折"。

**可以补充的知识星球故事类型：**

1. **早期粗糙带来的具体代价**
   - 不考虑续费功能，9 个月后真的要上线时遇到了什么技术难题？用户有没有因为续费体验差而流失？
   - 当机三次的具体场景：用户在做什么？他们的反馈是什么？团队怎么应对的？
   - 这些故事能让读者看到：粗糙是有代价的，但代价可控，关键是快速修正。

2. **功能粗糙上线后的快速迭代**
   - 有没有某个功能，第一版极其粗糙（比如只是个手动操作的流程），但上线后发现用户很需要，然后快速迭代完善的？
   - 比如：某个内容管理功能，第一版可能只是个简单的列表，但看到用户使用频率高，快速加上了搜索、筛选、排序等功能。

3. **用户对粗糙的容忍度**
   - 有没有遇到过用户抱怨某个功能太粗糙，但团队有意识地选择暂时不修？最后的结果是什么？
   - 或者相反：有没有某个地方团队觉得可以粗糙，但用户反馈强烈，必须立刻完善？
   - 这种故事能展示"判断粗糙边界"的真实过程。

4. **实验室功能的实际案例**
   - 有哪些功能放进了实验室，最后成功转正的？
   - 有哪些功能放进实验室，发现不行，直接砍掉的？
   - 早期采用者给了什么样的反馈？这些反馈如何影响了产品决策？

5. **从中间开始的具体例子**
   - 知识星球有没有某个功能，是先做出"最有表现力的核心部分"，验证可行后再补全其他部分的？
   - 比如：某个社交功能，可能先做出"能看到效果"的交互，用假数据验证体验，确认方向后再接真实后端。

6. **时间框约束的真实场景**
   - 有没有某次发布，是严格按照"三周内必须上线"这样的时间框来倒推规格的？
   - 过程中砍掉了哪些功能？事后证明这些被砍掉的功能是真的不重要，还是后来又补回来了？

**建议的写作方式：**

不要只是罗列"我们做了什么"，而要写出"为什么这么做 + 结果如何 + 学到了什么"。比如：

> 知识星球早期有个分享功能，产品经理希望支持分享到微信、微博、QQ、豆瓣等十几个平台。我们讨论后决定：先只做微信。理由很简单：我们的用户 90% 都用微信分享，其他平台加起来不到 10%。
>
> 第一版上线后，确实有用户问"为什么不能分享到微博"。我们记下来了，但没有立刻做。三个月后，数据显示只有不到 5% 的用户尝试用截图的方式分享到其他平台。我们决定继续不做。
>
> 半年后，微博那 5% 的需求自然消失了——因为微博本身的使用率在下降。如果当初花两周做了十几个平台的分享，这两周就浪费了。

这种故事有具体场景、有数据、有决策过程、有结果验证，读者能真正学到"怎么判断粗糙的边界"。

## Q3: 图片与表格建议

这一章的内容很适合用视觉化方式呈现，能大幅提升可读性和理解效率。

**建议的图片和表格：**

### 1. **粗糙 vs 完美的决策矩阵（表格）**

在"什么时候该粗糙，什么时候该完美"这个核心问题上，可以用表格呈现：

| 维度 | 粗糙 | 完美 |
|------|------|------|
| 阶段 | 验证期、探索期 | 增长期、成熟期 |
| 功能类型 | 非核心功能、实验性功能 | 核心功能、关键体验 |
| 用户类型 | 早期采用者 | 大众用户 |
| 成本 | 低成本快速验证 | 高成本深度打磨 |
| 目的 | 快速获得反馈 | 提升留存和口碑 |
| 风险 | 可接受的粗糙感 | 不可接受的低质量 |

这个表格能帮助读者快速建立判断框架。

### 2. **Shape Up 的三步法（流程图）**

把 Pitch → Setting the appetite → No gos 这三步用流程图呈现：

```
[提案：粗马克笔草图]
    ↓
[设定胃口：值得花多少时间？]
    ↓
[划定禁区：什么不做？]
    ↓
[开始执行：聚焦核心]
```

配上每一步的关键问题，比如：
- 提案：这个功能解决什么问题？
- 设定胃口：值得 1 天？1 周？还是 1 个月？
- 划定禁区：哪些相关需求暂时不做？

### 3. **MVP vs MEP 的对比图**

用两张示意图对比：

**MVP（最小可行产品）：**
```
[核心功能] → 能用就行 → 快速验证
```

**MEP（最小可进化产品）：**
```
[核心功能 + 进化能力] → 能用 + 能响应变化 → 持续进化
```

配上 Tesla 的例子：Roadster（科技感 + 加速）→ Model S（续航 + 豪华）→ Model 3（大众市场）→ Model Y（实用性）

这个图能让读者直观理解：不只是"能用"，还要"能进化"。

### 4. **时间框约束规格的示意图**

用甘特图或倒计时图呈现"先定时间，再定规格"的过程：

```
第 0 周：定目标：3 周后发布
第 1 周：列出规格（10 个功能）
第 2 周：发现做不完，砍掉 4 个
第 3 周：再砍掉 2 个，剩 4 个核心功能
第 3 周末：发布
```

这个图能让读者看到"动态调整规格"的过程。

### 5. **粗糙的边界在变化（趋势图）**

用时间轴呈现"粗糙的门槛在提高"：

```
1999 年 QQ：功能能用就行，界面可以很丑
2010 年移动互联网：界面要过得去，体验要流畅
2025 年 AI 时代：关键体验必须精致，否则用户不给第二次机会
```

配上每个时代的典型产品截图对比（如果版权允许）。

### 6. **从中间开始的示意图**

用图示展示"从中间开始"vs"从头开始"的区别：

**传统做法（从头开始）：**
```
[搭架构] → [写接口] → [做前端] → [联调] → [测试] → [验证创意]
```

**从中间开始：**
```
[验证创意（用假数据）] → [确认可行] → [补全其他部分]
```

这个图能让读者直观理解"先做最有表现力的部分"的价值。

### 7. **QQ 早期功能优先级（表格）**

把那道腾讯产品经理考题做成表格：

| 功能 | 是否选择 | 理由 |
|------|----------|------|
| 卡通头像 | ✓ | 认同感、想象空间 |
| 看谁在线 | ✓ | 最能粘住用户 |
| 很小的 .exe 文件 | ✓ | 技术栈自然带来 |
| 速度超快 0.5 秒反应 | ✓ | 技术栈自然带来 |
| 聊天室 | ✗ | 非核心 |

这个表格能让读者看到"抓核心"的具体判断过程。

### 8. **粗糙的代价（流程图）**

用流程图呈现"粗糙 → 遇到问题 → 快速修正"的循环：

```
知识星球不考虑续费
    ↓
9 个月后遇到问题
    ↓
快速补上续费功能
    ↓
用户续费率 XX%（如果有数据）
```

这种图能让读者看到：粗糙不是没有代价，而是代价可控。

**呈现建议：**

- 表格用于对比、框架、清单类内容
- 流程图用于展示时间线、决策过程
- 趋势图用于展示变化
- 示意图用于抽象概念的可视化

图片和表格不要只是"好看"，而要"能帮助读者快速理解复杂概念"。

## Q4: 整体评分

**评分：72/100**

这一章有丰富的案例，逻辑清晰，但存在几个明显问题：

### 扣分点 1：核心观点不够突出（-10 分）

这一章想说什么？读完后，读者能记住的核心观点是什么？

现在感觉是：各种案例堆砌，但缺乏一个贯穿全章的核心矛盾或核心观点。

**问题：**
- 标题是"粗糙"，但粗糙的本质是什么？是"快速试错"？是"聚焦核心"？还是"有选择的取舍"？
- 开头引用了两句话，一句是"如果你对第一版产品不感到尴尬，说明你发布得太晚了"，另一句是"简单是终极的复杂"。这两句话的关系是什么？它们如何统领全章？

**建议：**
- 在开头明确提出核心观点，比如："粗糙不是粗制滥造，而是在有限资源下的聚焦能力。关键在于知道什么是核心，什么可以暂时放下。"
- 然后用案例支撑这个核心观点，最后升华：粗糙的本质是"做出艰难的取舍，并为这个取舍负责"。

### 扣分点 2：案例过多，缺乏提炼（-8 分）

这一章有太多案例：Gumroad、知识星球、微信支付、Shape Up、轻读、EatVenture、腾讯早期故事、QQ、微信红包、Tesla。

案例多不是问题，问题是：
1. 案例之间的逻辑关系不够清晰
2. 每个案例讲完后，缺乏提炼和升华
3. 读者读完会觉得"信息量很大"，但"记不住什么"

**建议：**
- 给案例分类，比如：
  - 第一类：早期粗糙的成功案例（Gumroad、QQ）
  - 第二类：聚焦核心的案例（微信支付、知识星球快捷键）
  - 第三类：快速验证的案例（轻读、实验室功能）
  - 第四类：从中间开始的案例（Tony 的观点、Shape Up）
  - 第五类：粗糙的边界在变化（现在的粗糙更难、MEP）
- 每一类案例讲完后，用一两句话提炼核心方法论
- 最后用一个框架把所有方法论串起来

### 扣分点 3：缺乏具体的操作指南（-5 分）

这一章讲了很多"为什么要粗糙"，但"怎么粗糙"讲得不够具体。

比如：
- Shape Up 的三步法（Pitch、Setting the appetite、No gos）提到了，但没有展开，读者看完还是不知道怎么操作
- "从中间开始"提到了三个标准（必须是核心、必须小、必须有趣），但没有具体案例展示"怎么判断这三个标准"
- "用时间框约束规格"提到了"先定时间，再定规格"，但没有说"怎么砍功能"

**建议：**
- 选择 1-2 个方法，深入展开，给出具体的操作步骤
- 比如"用时间框约束规格"，可以写成：
  1. 定一个不可动摇的时间点（比如三周后）
  2. 列出所有想做的功能
  3. 估算每个功能的工作量
  4. 按重要性排序
  5. 从上往下选，选到时间用完为止
  6. 剩下的全部砍掉，写在"下一期"清单里
  7. 执行过程中，如果发现时间不够，继续砍

这种具体的操作指南，读者看完能立刻用。

### 扣分点 4：粗糙的代价讲得不够（-3 分）

这一章几乎都在讲粗糙的好处，但粗糙是有代价的：
- 知识星球当机三次，用户体验很差
- 不考虑续费，9 个月后要补坑
- Gumroad 手动给用户付款，规模到多大时这个方法就不行了？

这些代价没有充分展开，读者会觉得"粗糙是万能的"。

**建议：**
- 单独用一节讲"粗糙的代价"
- 展示具体的案例：粗糙到什么程度时，会失去用户？会错过机会？会造成技术债？
- 然后给出判断标准：什么样的粗糙是可接受的，什么样的粗糙会致命？

### 扣分点 5：结构松散，节奏不紧凑（-2 分）

这一章的结构是：
1. 开始可以简单粗糙一点
2. 微信支付的做减法
3. 知识星球里一个关于粗糙的例子
4. 用时间框约束规格
5. 小成本验证
6. 从中间开始
7. 快速 build 的力量
8. QQ 的例子
9. 现在的粗糙更难
10. 最小可进化产品

这个结构看起来是"想到哪儿写到哪儿"，缺乏内在逻辑。

**建议重新组织结构：**

**第一部分：粗糙的本质**
- 粗糙不是粗制滥造，而是聚焦核心
- 案例：QQ 只做 3 条协议、微信支付只绑银行卡

**第二部分：怎么粗糙**
- 方法 1：用时间框约束规格
- 方法 2：从中间开始
- 方法 3：小成本验证（实验室功能）
- 案例：知识星球、轻读、Shape Up

**第三部分：粗糙的边界**
- 什么时候该粗糙，什么时候不能粗糙
- 粗糙的代价
- 关键体验 vs 非关键功能
- 案例：现在的粗糙更难

**第四部分：从粗糙到进化**
- MVP vs MEP
- 快速 build 的力量
- 案例：Tesla、微信红包、QQ

这样的结构更紧凑，逻辑更清晰。

### 扣分点 6：有些案例展开过度（-2 分）

QQ 的案例占了全章的近 1/4 篇幅，展开得太详细了。虽然这些细节很有趣（比如企鹅公仔做错了、Kenny 写的统计脚本用了很多年），但偏离了"粗糙"这个核心主题。

**建议：**
- QQ 的案例聚焦在"11 周做出来"、"3 条协议"、"200K 客户端"、"一周发布 5-6 次"这些和"粗糙"直接相关的点
- 其他有趣的细节，可以放到脚注或"延伸阅读"

## Q5: 必要性审视

**不能砍。这一章很重要。**

理由如下：

### 1. 这一章解决的是"执行"环节最常见的痛点

创业者最常见的问题是：想太多，做太慢，追求完美，结果产品一直出不来。

这一章用大量案例告诉读者："可以粗糙一点"，这对很多人来说是解放性的。

特别是对于完美主义者、有大公司背景的人、技术背景的人，这一章能帮他们突破心理障碍。

### 2. 这一章和前面的章节形成对比

前面的章节讲"不一样"、讲"结构"，都在强调"要想清楚"。

但想清楚之后，怎么快速做出来？这一章给出了答案：聚焦核心，其他可以粗糙。

如果没有这一章，读者会觉得：你们又要想清楚，又要结构完整，又要不一样，我什么时候能开始做啊？

这一章告诉读者：想清楚不等于做完，可以快速做出最小版本。

### 3. 这一章的案例很丰富，有说服力

QQ、微信支付、微信红包、知识星球，这些案例都很有分量。

特别是 QQ 的故事，细节丰富，能让读者看到"粗糙"在真实世界里是怎么运作的。

### 4. 这一章提出了重要的方法论

- Shape Up 的三步法
- 从中间开始
- 用时间框约束规格
- 实验室功能
- MEP vs MVP

这些都是可操作的方法论，读者看完能用。

### 但是，这一章需要大幅改进

虽然不能砍，但这一章现在的状态只能算"及格"，远远没有达到"优秀"。

**核心问题是：案例太多，提炼不够。**

读者读完会觉得"信息量很大"，但"记不住什么"。

**改进方向：**

1. **明确核心观点**：粗糙的本质是什么？开头就要说清楚。
