# 《小落点》审阅意见

> 全书第 8 章 / 共 40 章

## Q1: 共识与非共识

**常识部分**
"小而美"、"从小做起"、"先做一个简单的 MVP"，这些观点已经是创业圈的共识了。尤其是 GitHub 的案例，几乎成了教科书级别的励志故事。

**非共识观点建议**

这一章缺少一个反向思考的角度：什么时候小落点是个陷阱？

建议增加以下非共识内容：

1. **小落点的陷阱：太小的市场根本做不大**
   - 不是所有小落点都能变大。有些市场就是小到头了
   - Pinboard 的案例其实就是证明：2.9 万用户到峰值，年收入 25 万，之后就只能看着它慢慢衰退
   - 需要补充判断标准：什么样的小落点有可能变大，什么样的只能养活自己

2. **小落点的时代局限性**
   - GitHub 能做起来，跟那个时代开发者工具荒漠有关。今天再做一个"开发者的某某工具"，已经很难了
   - 2008-2012 年是移动互联网爆发前夜，Web 应用迭代快，试错成本低。今天呢？App 生态成熟，试错成本高得多
   - 建议补充：不同时代，小落点的策略完全不同。现在适合做小落点的领域在哪里？

3. **一人公司的残酷真相**
   - Maciej 那封邮件已经说得很清楚了："汤会凉着端上来，或者根本不上菜"
   - 一人公司意味着放弃增长，接受衰退，这是个很大的心理代价
   - Gary Brewer 能做到 2260 万美元，是因为他选的赛道（销售线索生成）自带高客单价。不是所有一人项目都能这样

4. **与苏世民观点的真正矛盾**
   - 这一章试图反驳苏世民的"做大事和做小事一样难"，但没有说清楚真正的矛盾点
   - 真正的矛盾不是"大小"，而是"资源匹配度"
   - 苏世民说的对，但前提是你得有那个资源。小团队选小落点，本质上是资源约束下的理性选择，不是主动战略
   - 如果你有资源，直接做大事确实效率更高

**结论**
这一章太温和了。全是"小落点很好"的论述，但没有告诉读者：
- 什么时候应该选小落点
- 什么时候小落点会害死你
- 选了小落点之后，如何判断要不要升级

## Q2: 知识星球故事建议

知识星球本身就是个小落点的典型案例，但这一章只提了两句，完全没有展开。建议补充以下内容：

**1. 知识星球早期的小落点策略**
- 2016 年最早叫"小密圈"，只做一件事：帮有粉丝基础的大 V 做付费社群
- 不跟微信群竞争，不跟 QQ 群竞争，不做免费社群，只做付费深度社群
- 这个落点有多小？当时几乎所有人都在说"内容应该免费"，知识付费还没起来
- 吴鲁加是怎么找到这个切口的？他观察到了什么？

**2. 从小落点到生态的演化**
- 小密圈改名知识星球，背后是从"工具"到"平台"的转变
- 什么时候开始做内容沉淀功能？什么时候开始做精华整理？什么时候开始做星球推荐？
- 这些演化都是在小落点站稳之后，慢慢长出来的

**3. 知识星球的取舍故事**
- 一定有很多用户要求做即时通讯，要求做语音通话，要求做直播。为什么都没做？
- 什么功能坚决不做？为什么不做？这些取舍的背后逻辑是什么？
- 知识星球团队多大？为什么能保持小团队？

**4. 对比案例：其他知识付费产品的"大落点"失败**
- 很多知识付费产品一开始就想做大平台，最后都死了
- 可以对比得到、樊登读书会这些"大落点"产品，他们是怎么做的？
- 为什么知识星球选择了小落点，反而活得更好？

**建议**
把知识星球的完整故事补充进来，作为一个贯穿全章的主线案例。现在 GitHub、Pinboard、BuiltWith 都是国外案例，读者可能有距离感。知识星球是你自己做的产品，应该讲得更深入、更真实。

## Q4: 整体评分

**评分：72/100**

**扣分说明**

**1. 案例堆砌，缺少穿透力（-10分）**
- GitHub、Pinboard、BuiltWith 三个案例讲得很细，但缺少一个统一的分析框架
- 读完之后只知道"小落点能成功"，但不知道"我应该怎么选小落点"
- 案例之间的对比分析太少。为什么 BuiltWith 能做到 2260 万，Pinboard 只能 22 万？这个对比更有价值

**2. 观点单一，缺少反向思考（-8分）**
- 全章都在说"小落点好"，但没有说"什么时候小落点是陷阱"
- 苏世民的观点只用来当靶子打，没有真正思考他说的对在哪里
- 读者看完会觉得"小落点永远是对的"，这是误导

**3. 知识星球案例太浅（-5分）**
- 这是你自己的产品，应该讲得最深入，结果只提了两句
- 知识星球从 2016 到现在的演化路径，正好可以完整展示"小落点如何变大"
- 浪费了最好的素材

**4. 缺少操作指南（-3分）**
- 最后"选择：什么值得做"这一节太简略
- "做出不一样的东西"、"愿意长期做"这两个标准太虚，不够具体
- 需要一个更清晰的判断框架：什么样的小落点值得做，什么样的不值得

**5. 文字表达有重复（-2分）**
- "更容易活下来"、"巨头看不上"、"能做到极致"、"小可以变大"这四个小节，逻辑有重叠
- 可以合并优化，腾出篇幅讲更深入的内容

**加分项**
- 案例选择很好，GitHub、Pinboard、BuiltWith 都是经典案例（+3分）
- Maciej 那封邮件的引用特别真实，有画面感（+2分）
- 最后"台阶"的比喻很形象（+1分）

## Q5: 必要性审视

**不能砍。**

这一章在全书结构中承上启下，位置关键。

**为什么不能砍**

1. **承接前面的理论，落到具体方法**
   - 前面讲分形思维、黑客思维，很抽象
   - 第 8 章开始讲"如何找到第一个切入点"，小落点是第一个具体方法
   - 砍掉这章，理论和实践之间就断了

2. **回应全书核心命题：小团队怎么做**
   - 全书叫《夹缝生长》，核心就是"小"
   - 小落点是"小"的第一个具体体现
   - 不讲清楚"落点要小"，后面的章节都没有根基

3. **为后面章节铺垫**
   - 第 9 章"产品金字塔"会讲如何从小落点逐步扩大
   - 第三部分"具体执行"所有章节，都是基于"你已经选了一个小落点"这个前提
   - 砍掉第 8 章，后面的章节都不知道在讲什么

**但是，这一章需要大改**

问题不是"要不要这一章"，而是"这一章现在写得不够好"。

**建议修改方向**

1. **补充非共识观点**：小落点什么时候是陷阱
2. **深挖知识星球案例**：把自己的产品讲透
3. **增加决策框架**：给读者一个可操作的判断标准
4. **补充对比分析**：为什么 BuiltWith 和 Pinboard 差距这么大

改完之后，这一章会从 72 分变成 85 分以上。

---

**总结**

这是一章"安全"的稿子。案例充足，逻辑清晰，符合创业圈的主流观点。但也因此缺少锋芒。

读者看完会点头认同，但不会感到意外，不会有"原来还可以这样想"的冲击感。

建议增加反向思考，把知识星球案例讲深，把决策框架讲透。让这一章不只是说"小落点好"，而是说清楚"什么时候选小落点，什么时候不选，选了之后怎么判断要不要升级"。

这样才配得上全书第 8 章的位置。
