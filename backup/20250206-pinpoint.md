# 小落点

> Small is beautiful.
> 小即是美。
>
> —— E.F. Schumacher

## Github 的故事

2008 年 4 月 10 日，GitHub 正式上线。

三个创始人，Tom Preston-Werner、Chris Wanstrath、PJ Hyett，除了 PJ 之外都是大学辍学。他们凑了几百美金开始，靠给 Engine Yard 和 Rackspace 打广告换免费服务器。

上线前四个月，他们只做了一件事：让程序员能方便地分享代码。

![2007 年的 GitHub 早期版本](img/pinpoint-01.jpg)

创始人说："我们都喜欢 git，但没有合适的方式跟别人分享代码。"就这么简单的需求，成了 GitHub 的起点。他们把 GitHub 描述成“程序员的维基百科”：你能编辑文件，看谁改了什么，查看旧版本，在世界任何地方访问——只不过你操作的是代码，而不是百科全书。

商业模式也简单到极致：公开代码库免费，私有代码库收费。

我很喜欢创始人在早期说的两段话。

"我们专注于把事情做完。很幸运我们做的主要是 Web 应用，这意味着可以超快速迭代。我们学到的是，现在就发布，然后根据用户使用情况修复，比让它永远停留在开发阶段要好得多。直接发布就对了。"

还有一段："盈利之前和之后都要关心每一分钱怎么花。专注于重要的事，别浪费时间没完没了地重新设计网站或折腾新技术。以后有的是时间做这些。"

这两段话给我很深的触动。

- Web 应用可以快速迭代，哪怕现在，我们也应该多用 Web 版做 MVP
- 发布胜过一切，不要等完美版本
- 要很在意现金流
- 别太在意重构和追求新技术，先找到产品与市场的匹配

GitHub 前四年没拿过一分钱融资，一直盈利。2008 年收入 100 万美元，2009 年 200 万，2010 年 900 万。

2012 年 7 月，以 7.5 亿估值从 a16z 融资 1 亿美金，这是他们第一次融资。

到 2018 年微软 75 亿美金收购时，GitHub 年收入 3 亿美金，有 2800 万用户。2022 年，年收入突破 10 亿美金，活跃用户 9000 万。

再看用户和代码库的增长。2008 年上线时，有 3.8 万个代码库。2013 年底，280 万用户，460 万代码库。2018 年，3100 万用户，9600 万代码库。2023 年，1 亿开发者，2.8 亿代码库。

![GitHub 开发者与代码库增长曲线](img/pinpoint-02.png)

这个增长曲线里藏着一个关键时机：开发者数量的爆发。

2011 年 8 月，Marc Andreessen 写了那篇著名的文章“软件正在吞噬世界”。接下来的十多年，全球开发者数量爆炸式增长。GitHub 踩中了这波浪潮，在持续增长的市场里，做了多年有复利的事。

现在 AI 来了，“人人都是开发者”的时代可能真的要到了。GitHub 又站在了新一轮增长的起点上。

回头看，GitHub 解决的问题其实不复杂。

对开发者来说，解决了大规模远程协作、交流代码的需求，也让 GitHub 成了开发者对外的名片。对企业来说，提供了存储代码的 SaaS 服务。

落点很小，但长期运行，时机又好，就等到了两次爆发：程序员数量的爆发，和 AI 带来的“人人都是开发者”。

## 变大的梦想

《苏世民：我的经验与教训》里有个观点：做大事和做小事的难易程度是一样的。所以要选择一个值得追求的宏伟目标，让回报与你的努力相匹配。

他说：“实现大目标和小目标的难度相差无几，唯一的区别在于目标越大，影响力也越大。人的精力有限，既然每次只能聚焦一项事业，就应该选择一个真正值得努力的目标。”

他还提出了对创业项目的三项基本测试。

- 你的设想必须足够宏大，值得你全身心投入，有潜力发展成规模巨大的企业
- 产品或服务应该是独一无二的。当人们看到你提供的东西时，应该对自己说："我的天哪，我需要这个，我一直在等这个东西。"如果你的产品没有让人喜出望外、拍案叫绝，就是在浪费时间
- 时机必须是正确的。这个世界不喜欢开拓者，进入市场太早，失败风险很高。你瞄准的市场应该有足够的发展势头

如果通过这三项测试，你就拥有了一个具有巨大潜力的企业。能提供独特的产品或服务，也能在合适的时间进入市场。然后，你必须做好准备，迎接痛苦。新事物的诞生必然伴随着阵痛。

这套理论听起来很对，但有两个前提。

一是你得有苏世民那样的资源。他创办黑石集团时，已经是雷曼兄弟的董事总经理，有人脉、有经验、有资金渠道。他说"做大事和做小事一样难"，是因为对他来说，确实一样难。

更关键的是，时代不一样了。

苏世民创业是 1985 年。那时候大市场还有大量空白，先发优势巨大，做大事确实有可能一步到位。

但今天的互联网市场，格局已经定了。搜索有百度，社交有微信，电商有淘宝京东拼多多，短视频有抖音快手。每一个大市场都有巨头把守，基础设施早已成熟。

在这个时代，想做大事，要么你有巨头级别的资源，正面突破；要么你先找到一条夹缝，慢慢生长。

对没有特别资源的小团队来说，小落点是更现实的选择。

而小落点，恰恰是最容易找到夹缝的方式。市场够小，巨头看不上；需求够具体，你能做到极致；成本够低，你能活下来等待机会。

这不是退而求其次，这是这个时代小产品的生存法则。

## 小落点是夹缝生存的策略

**更容易活下来**

成本低，现金流压力小，更快达到盈利。不依赖融资，就不用被资本节奏绑架。

GitHub 前四年没融资，一直盈利。2008 年收入 100 万，2009 年 200 万，2010 年 900 万。等到 2012 年融资时，他们已经是一家健康的生意，融资是锦上添花，不是续命。

知识星球从一开始就盈利为先，保持小团队高人效。不追求规模，不追求融资，所以能按自己的节奏发展，不用担心资金链断裂。

小落点让你有时间试错，有机会等到真正的机会。

**巨头看不上，给你时间**

市场太小，大公司不屑于做。等你做大了，已经建立了护城河。

微信、QQ 都有群功能，但“付费深度社群”这个切口太小，巨头不会专门做。用户量不够大，变现不够快，不符合他们的战略。

这给了知识星球时间。可以慢慢打磨产品，积累用户，建立品牌。等巨头反应过来，你已经在这个小市场里站稳了。

GitHub 也是。2008 年时，大公司在做大而全的代码托管方案，没人在意“给开源项目提供免费托管”这个小需求。等他们看到 GitHub 起来了，已经晚了。

**能把一件事做到极致**

用户群明确，需求聚焦，不会被多方需求拉扯。

知识星球早期只服务“有粉丝基础、希望深度服务铁杆粉丝的内容创作者”，而不是“所有想做社群的人”。这让产品能深入打磨核心功能：付费订阅、内容沉淀、精华整理。

如果一开始就想做大而全的社群平台，就要同时满足企业团队协作、兴趣社交、粉丝运营等多种场景。每个场景都做不透，最后变成四不像。

专注一个小落点，反而能做出差异化。

**小可以变大，但大很难变小**

从小做大是自然演化，从大做小是战略收缩，后者很难做到。

GitHub 从“开源项目代码托管”起步，慢慢扩展到私有仓库、企业服务、AI 工具。每一步都是在前一步站稳之后，自然生长出来的。

知识星球也是。最初可能目标用户更窄，后来随着知识付费兴起，自然扩大到更多领域。但核心始终是“深度服务付费用户”。

反过来，如果一开始就想做大平台，最后发现做不动，要收缩到小市场，会非常痛苦。产品功能砍不掉，团队缩不回去，成本降不下来。

小落点给你演化的空间。

## 一人项目

小落点的极致，是一人公司。

不追求规模，不追求增长，只做一件简单的事，让它稳定运行多年，养活自己。这听起来像是退而求其次，但对很多人来说，这恰恰是最理想的状态。

Pinboard 和 BuiltWith 是两个典型例子。

### Pinboard

2010 年，程序员 Maciej Ceglowski 做了一个书签管理工具叫 Pinboard。

极简。用户可以保存链接，打标签，搜索，导出。就这些功能，没有社交，没有推荐，没有花哨的界面。很多人觉得简陋，但对需要的人来说，够用。

商业模式也简单：年付 22 美元。

从 2010 年到现在，Maciej 一个人运营了 14 年。2017 年活跃用户达到峰值，2.9 万人，年收入 25.9 万美元。此后用户和收入逐年下滑，2024 年活跃用户 1.3 万，收入 22.8 万美元。

我是 Pinboard 早期的付费用户。

用了几年后放弃，因为它对中文网站支持很差。比如微信公众号的标题，长期获取不到，给作者写信也没回应。后来辗转试了 Pocket、Instapaper 这些产品，都不喜欢，最终放弃了书签收藏这件事。

但 Pinboard 给我留下很深印象：只需要一万多活跃用户，就能有体面的收入。一个人，不用管理团队，不用融资，不用开会，只要保持产品运行，就能养活自己。

2021 年，我收到 Maciej 的邮件。他说自己一个人运营越来越吃力，想把早期的一次性付费用户转成年付。

邮件里有段话很有意思：“我的工作就像单人经营一家老城堡里的餐厅。很酷，很有趣，氛围也很好。但偶尔汤会凉着端上来，或者根本不上菜，因为我得去厨房赶走一群蝙蝠，或者修一根塌了的横梁，客人只能坐着等。这对我和客人来说都不好玩。”

他还说：“传统上，处在我这个位置的公司会拿外部投资，找 VC 讲个增长故事，或者转向新市场。但我更希望这个网站保持独立、固执、自给自足。我认真对待这份工作，希望它能运营到我职业生涯结束。”

这就是一人项目的样子。

做不大，增长乏力，甚至在缓慢衰退。但足够简单，成本可控，能稳定运行多年。创始人不用为增长焦虑，不用为融资操心，可以按自己的节奏生活。

当然，不迭代也有代价。用户会流失，市场会萎缩。但只要基数够支撑生活，就能一直做下去。

Pinboard 还在运行。Maciej 还是一个人。

### BuiltWith

BuiltWith 更酷一些。同样是一个人，但 2024 年年收入 2260 万美元。

2007 年，澳大利亚程序员 Gary Brewer 还在上班。他有个习惯：看到新网站，就查看源代码，看它用了什么技术。这是程序员的好奇心。

但每次都要手动查，很麻烦。市面上也没有工具。他想，为什么不做一个自动化工具？

就这样，BuiltWith 诞生了。

最开始只是业余项目，下班后写代码，持续开发了四年。功能很简单：输入网址，告诉你这个网站用了什么技术——服务器、CMS、分析工具、广告网络等等。

Gary 没想过赚钱。但用户很快发现了商业价值。

比如，一家邮件营销公司，可以用 BuiltWith 找到所有用竞争对手产品（如 Mailchimp）的网站列表。这就是一份高质量的潜在客户名单。销售可以直接按这份名单去推销。

这个场景成了 BuiltWith 最核心的商业模式：销售线索生成。

付费订阅的需求很快来了。Gary 还在犹豫要不要全职创业时，BuiltWith 已经每月 4 万美元收入了。

Gary 极度厌恶管理。他说过，只有月收入到 10 万美元，才会考虑招第一个程序员。他想把所有时间花在产品开发上，不想处理人事问题。

所以他把自动化做到极致。大部分客户问题通过知识库在 10 秒内解决。新问题出现，他就录个视频，以后复用。

VC 的电话每周都有，Gary 全都忽略。他觉得跟 VC 聊天是浪费时间。只要产品好，增长是自然的事。有意思的是，后来有 VC 支持的竞争对手出现，反而帮 BuiltWith 教育了市场，收入增长更快了。

从 2011 年月收入 4 万美元，到 2017 年年收入 1400 万美元，再到 2024 年 2260 万美元。BuiltWith 还是 Gary 一个人。

这个例子说明：小产品、小落点，也能做到很高的收入。关键是找到真实的需求，用自动化控制成本，不被增长和融资的焦虑绑架。

## 选择：什么值得做

GitHub 从代码托管起步，Pinboard 只做书签管理，BuiltWith 只看网站技术栈。都是小得不能再小的落点。

但小落点有个好处：台阶低。

爬山的时候，台阶如果是恰当的高度，你能一步一步往上走。如果台阶太高，第一步就耗尽力气，或者根本上不去。

很多人一上来就想做平台、做生态、做大事。这就像给自己设计了很高的台阶。除非你天赋异禀，就是个巨人，否则很难走远。

小落点不是没有野心，而是先让自己活下来，有时间试错，有机会等待。

GitHub 等了十年，等到开发者数量爆发。现在又等到了 AI 时代。Pinboard 和 BuiltWith 选择不等，就做一个能养活自己的小生意，也活得很好。

那怎么选择自己的小落点？

我自己有两个标准。

一是做出不一样的东西。不只是抄，不只是做"又一个某某"。你要能说清楚，相比现有方案，你在哪里更好。

GitHub 比 SourceForge 好在哪里？社交化、更简单、对开源友好。Pinboard 比 Delicious 好在哪里？更快、更可靠、不会倒闭。BuiltWith 是第一个做这件事的，没有竞品。

二是愿意长期做，有长期迭代空间。

不是反应特别敏捷的人，就不要做太短线的事。选一个自己真正喜欢、愿意探索十年的方向，用“笨拙但坚持”，用时间换价值。

知识星球也是这样。2016 年开始做，到现在快十年了。没做大，但一直在做。

小落点的价值，不在于一开始就能看到终点，而在于给你足够的时间，慢慢走到终点。
